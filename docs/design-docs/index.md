# design-docs/index.md — 설계 결정 인덱스

GoGo의 모든 설계 결정을 한 곳에서 찾을 수 있는 인덱스입니다.
새로운 결정이 내려지면 이 파일을 업데이트하고, 상세 내용은 해당 파일에 기록하세요.

**설계 결정의 두 가지 종류:**
- **ADR (Architecture Decision Record)**: 기술/아키텍처 결정. 번복 가능, 상황에 따라 변경됨
- **Core Belief**: 제품 철학. 바뀌면 제품 자체가 바뀌는 것. 신중하게 다뤄야 함

## Core Beliefs

제품의 근본 신념입니다. 변경 시 팀 전원 동의 필요. 상세 내용: `docs/design-docs/core-beliefs.md`

| # | 신념 | 핵심 한 줄 |
|---|------|-----------|
| 1 | 참여의 문턱은 0이어야 한다 | 초대 링크 + 닉네임만으로 즉시 참여 |
| 2 | 장소가 중심이다, 사람이 아니라 | 소셜 그래프가 아닌 장소 중심 설계 |
| 3 | 작은 그룹을 위해 만들어졌다 | 2~10명의 친밀한 그룹, 복잡한 권한 없음 |
| 4 | 기록은 실제 방문으로 완성된다 | WANT_TO_GO → VISITED 여정 |
| 5 | 결정을 빠르게 돕는다, 대신하지 않는다 | 알고리즘 추천 없음, 사람이 결정 |
| 6 | 복잡함은 제품이 지는 것이다 | 기능 추가보다 매끄러움 우선 |

## 결정이 필요한 미결 사항

아직 공식 ADR로 기록되지 않았지만 방향을 정해야 하는 것들입니다.

| 주제 | 맥락 | 관련 문서 |
|------|------|----------|
| 닉네임 → 계정 전환 시점과 방법 | 현재 그룹/투표가 닉네임 기반, 계정과 연결 안 됨 | `ARCH-001` |
| Flyway 도입 시점 | 스키마 변경 이력 없음, 수동 SQL 실행 중 | `REL-005` |

| 이미지 직접 업로드 vs URL 스크래핑 유지 | 현재는 OG 태그 스크래핑만 | `ADR-007` |

## 새 ADR 작성 방법

중요한 기술 결정을 내렸을 때, 아래 양식으로 `docs/DESIGN.md`에 추가하고 이 인덱스를 업데이트하세요.

```markdown
### ADR-NNN: [결정 제목]

**결정**: 한 문장으로 무엇을 결정했는지.

**이유**: 왜 이 결정을 했는지. 다른 선택지와 비교한 이유.

**트레이드오프**: 이 결정의 단점 또는 감수한 것들.

**파생 규칙**: 이 결정으로부터 나오는 하위 규칙들.
```

> 결정의 "무엇"보다 "왜"가 더 중요합니다. 나중에 이 결정을 번복할지 판단할 때 이유가 핵심입니다.
